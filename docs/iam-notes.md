# IAM Module Notes

## Overview

AWS Identity and Access Management (IAM) controls **who** can access AWS resources and **what** actions they are allowed to perform.

For this project, our EC2 instance needs permission to create and manage EBS snapshots without storing AWS access keys on the instance.

The architecture looks like this:

```
EC2 Instance
      │
      ▼
IAM Instance Profile
      │
      ▼
IAM Role
      │
      ▼
IAM Policy
      │
      ▼
AWS APIs (EC2 / EBS)
```

---

# Terraform: Data vs Resource

Terraform has two important concepts:

## Resource

A **resource** creates and manages infrastructure.

Example:

```hcl
resource "aws_iam_role" "this" {
  ...
}
```

Terraform will:

- Create the resource
- Update it
- Delete it
- Track it in Terraform state

Think of it as:

> "Create something inside AWS."

---

## Data

A **data source** does **not** create infrastructure.

Instead, it either:

- Reads existing information from AWS
- Generates data that another resource can use

Example:

```hcl
data "aws_iam_policy_document" "assume_role" {
    ...
}
```

This does **not** create anything inside AWS.

It simply generates a JSON policy document.

Think of it as:

> "Prepare or read information."

# IAM Module Breakdown

---

## 1. EC2 Assume Role Policy

```hcl
data "aws_iam_policy_document" "assume_role" {

  statement {

    effect = "Allow"

    principals {

      type = "Service"

      identifiers = [
        "ec2.amazonaws.com"
      ]

    }

    actions = [
      "sts:AssumeRole"
    ]

  }

}
```

### Purpose

This generates the **Trust Policy** for the IAM Role.

Nothing is created inside AWS.

Terraform simply generates JSON like:

```json
{
  "Version":"2012-10-17",
  "Statement":[
    {
      "Effect":"Allow",
      "Principal":{
        "Service":"ec2.amazonaws.com"
      },
      "Action":"sts:AssumeRole"
    }
  ]
}
```

---

## Why use `aws_iam_policy_document`?

Instead of manually writing JSON:

```hcl
assume_role_policy = <<EOF
{
   ...
}
EOF
```

Terraform generates valid JSON automatically.

Advantages:

- Easier to read
- Validates syntax
- Less error-prone
- Supports variables
- Easier to maintain

---

# 2. IAM Role

```hcl
resource "aws_iam_role" "this" {

  name = var.role_name

  assume_role_policy = data.aws_iam_policy_document.assume_role.json

}
```

### Purpose

Creates an IAM Role.

AWS Console:

```
IAM
 └── Roles
      └── dev-ec2-role
```

The role uses the Trust Policy generated above.

---

## Trust Policy

Every IAM Role has a **Trust Policy**.

It answers:

> **Who is allowed to use this role?**

Examples:

- EC2
- Lambda
- ECS
- GitHub Actions

Our project allows:

```
ec2.amazonaws.com
```

This means:

Only EC2 instances can assume this role.

---

# Trust Policy vs Permission Policy

These are completely different.

## Trust Policy

Answers:

> Who can assume this role?

Example:

```
EC2
```

---

## Permission Policy

Answers:

> What is this role allowed to do?

Example:

- Create Snapshots
- Describe Volumes
- Describe Instances

---

# 3. IAM Permission Policy

```hcl
data "aws_iam_policy_document" "ebs_snapshot" {

    statement {

        effect = "Allow"

        actions = [

            "ec2:DescribeInstances",
            "ec2:DescribeVolumes",
            "ec2:DescribeSnapshots",
            "ec2:CreateSnapshot",
            "ec2:CreateTags"

        ]

        resources = ["*"]

    }

}
```

### Purpose

Again, this generates JSON.

Nothing is created in AWS.

Generated JSON looks like:

```json
{
    "Statement":[
        {
            "Effect":"Allow",
            "Action":[
                "ec2:CreateSnapshot",
                "ec2:DescribeVolumes"
            ],
            "Resource":"*"
        }
    ]
}
```

---

# 4. IAM Policy

```hcl
resource "aws_iam_policy" "this" {

    name = var.policy_name

    policy = data.aws_iam_policy_document.ebs_snapshot.json

}
```

### Purpose

Creates an IAM Policy.

AWS Console:

```
IAM
 └── Policies
      └── dev-ebs-snapshot-policy
```

Terraform uses the JSON generated by the previous data block.

---

## Why create a separate IAM Policy?

Policies are reusable.

Example:

```
EC2
Lambda
ECS
```

All three can attach the same policy.

Instead of duplicating permissions, AWS encourages reusable policies.

---

# 5. Role Policy Attachment

```hcl
resource "aws_iam_role_policy_attachment" "this" {

    role = aws_iam_role.this.name

    policy_arn = aws_iam_policy.this.arn

}
```

### Purpose

Attaches the IAM Policy to the IAM Role.

Before attachment:

```
IAM Role

IAM Policy
```

They exist independently.

After attachment:

```
IAM Role
      │
      ▼
IAM Policy
```

Without this attachment, the role has **no permissions**.

---

# 6. Instance Profile

```hcl
resource "aws_iam_instance_profile" "this" {

    name = var.instance_profile_name

    role = aws_iam_role.this.name

}
```

### Purpose

Creates an Instance Profile.

This is one of the most confusing AWS IAM resources.

EC2 **cannot attach an IAM Role directly**.

AWS requires:

```
EC2
    │
    ▼
Instance Profile
    │
    ▼
IAM Role
```

The Instance Profile acts as a container for the IAM Role.

---

# Why does AWS use Instance Profiles?

When launching an EC2 instance, AWS expects:

```hcl
iam_instance_profile = module.iam.instance_profile_name
```

Not:

```hcl
role_name = ...
```

AWS internally maps:

```
EC2
      │
      ▼
Instance Profile
      │
      ▼
IAM Role
```

# Summary

| Block | Type | Creates AWS Resource? | Purpose |
|------|------|-----------------------|---------|
| `aws_iam_policy_document (assume_role)` | Data | ❌ | Generates Trust Policy JSON |
| `aws_iam_role` | Resource | ✅ | Creates IAM Role |
| `aws_iam_policy_document (ebs_snapshot)` | Data | ❌ | Generates Permission Policy JSON |
| `aws_iam_policy` | Resource | ✅ | Creates IAM Policy |
| `aws_iam_role_policy_attachment` | Resource | ✅ | Attaches Policy to Role |
| `aws_iam_instance_profile` | Resource | ✅ | Allows EC2 to use the IAM Role |

---

# Key Takeaways

- **Data sources** do not create infrastructure.
- **Resources** create and manage AWS infrastructure.
- Every IAM Role requires a **Trust Policy**.
- Permissions are granted through **IAM Policies**.
- Policies must be attached to Roles.
- EC2 uses an **Instance Profile**, not the Role directly.
- Terraform's `aws_iam_policy_document` is preferred over writing raw JSON because it is readable, validated, and easier to maintain.
